# 0001. Gold layer star schema

## Status

Accepted

## Context

The Silver layer produces validated, deduplicated, referentially-consistent data for five
entities: Customer, Agent, Policy, Claim, and Payment. Each is keyed by a UUID and stored as a
flat table matching its pydantic model.

The Gold layer needs to reshape this into a star schema that is easy and fast to query from
Power BI and Microsoft Fabric: few joins, integer surrogate keys, denormalized dimensions, and
fact tables that carry only measures, degenerate attributes, and foreign keys.

## Decision

### Surrogate keys

* Every dimension and fact table gets its own integer surrogate key, generated by sorting rows
  by their natural (UUID) key and assigning sequential integers starting at 1. This keeps
  surrogate keys stable and reproducible for a given seed/dataset, without requiring a separate
  key-management service.
* `dim_date` is the one exception: its surrogate key is the date itself in `YYYYMMDD` integer
  form (e.g. `20240115`). This is a standard convention — sortable, human-readable, and
  computable directly from any date without a lookup.
* Every table also retains its Silver-layer UUID as a natural/degenerate key, so a Gold row can
  always be traced back to Silver and Bronze.

### Dimensions

| Table | Grain | Surrogate key | Notable attributes |
|---|---|---|---|
| `dim_date` | One row per calendar date | `date_key` (`YYYYMMDD` int) | year, quarter, month, month_name, day, day_of_week, day_name, is_weekend |
| `dim_customer` | One row per customer | `customer_key` (int) | name, contact info, address — flat, denormalized |
| `dim_agent` | One row per agent | `agent_key` (int) | name, agency name, license number/state |

`dim_date` is generated directly (a contiguous calendar range covering all transactional dates),
independent of any Silver data. `dim_customer` and `dim_agent` are built from their Silver
counterparts.

### Facts

| Table | Grain | Surrogate key | Foreign keys | Measures |
|---|---|---|---|---|
| `fact_policy` | One row per policy | `policy_key` (int) | `customer_key`, `agent_key`, `effective_date_key`, `expiration_date_key` | `premium_amount` |
| `fact_claim` | One row per claim | `claim_key` (int) | `policy_key`, `date_of_loss_key`, `report_date_key` | `claim_amount`, `paid_amount` |
| `fact_payment` | One row per payment | `payment_key` (int) | `policy_key`, `payment_date_key` | `amount` |

Degenerate dimension attributes (`policy_number`, `policy_type`, `status`, `claim_number`,
`claim_type`, `payment_number`, `payment_method`, etc.) are kept directly on their fact table
rather than broken out into separate dimensions — with data at this volume, junk dimensions add
join overhead without a meaningful storage or usability win.

### Fact-to-fact reference: `fact_claim` and `fact_payment` → `fact_policy`

`fact_claim` and `fact_payment` reference `fact_policy.policy_key` directly, rather than through
a separate `dim_policy`. This is a deliberate deviation from strict Kimball style (dimensions and
facts are usually kept distinct), chosen because:

* Policy already carries a genuine measure (`premium_amount`), so it is legitimately a fact table,
  not just a dimension.
* Claims and payments are naturally children of a policy (header/detail relationship), and Power
  BI's Direct Lake / composite models handle fact-to-fact single-direction relationships on an
  integer key without issue.
* It avoids maintaining the same policy attributes in two places (`fact_policy` and a would-be
  `dim_policy`).

The tradeoff is that this isn't a pure star schema — see Future Considerations.

## Consequences

* All Gold tables are Parquet files under `output/gold/`, one file per table, matching the
  Bronze/Silver convention.
* Every transformation needed to build a table (dimension or fact) depends only on Silver data
  and, where relevant, on dimensions/facts already built earlier in the same run — enforced by
  build order in the `transform` command (`dim_date`, `dim_customer`, `dim_agent`, then
  `fact_policy`, then `fact_claim`/`fact_payment`).
* Surrogate keys are deterministic for a given seed, so Gold output is reproducible across runs,
  matching the reproducibility guarantee already provided by the seeded generators.

## Future Considerations

* If a `dim_policy` ever becomes necessary (e.g. to support policy attribute changes over time
  independent of `fact_policy`), it can be introduced without changing `fact_claim`/
  `fact_payment`'s grain — only their foreign key target would change.
* Slowly changing dimensions (SCD Type 2) for `dim_customer` and `dim_agent` are out of scope for
  now; every Gold build currently reflects the latest Silver snapshot only.
